# 技術Q&A - 2026-03-05

### SecretKeySpec とは何か？

`SecretKeySpec` は Java の `javax.crypto` パッケージが提供するクラスで、バイト配列から対称鍵（SecretKey）オブジェクトを生成するアダプター。

AES などの対称暗号アルゴリズムを使う際、Java の暗号化 API（`Cipher` クラス等）は生のバイト配列ではなく `SecretKey` インターフェースの実装を要求するため、このクラスでラップする。

```java
new SecretKeySpec(KeyBytes, "AES")
//                ↑           ↑
//         鍵のバイト配列   アルゴリズム名
```

**TokenEncryptionService でのデータフロー:**

```
application.yml の Base64文字列
        ↓ Base64.getDecoder().decode()
    byte[] (生のバイト配列 = 実際の鍵データ)
        ↓ new SecretKeySpec(KeyBytes, "AES")
    SecretKey オブジェクト (AESで使える鍵)
        ↓
    Cipher.init() に渡して暗号化・復号に使う
```

設定ファイルにバイナリを直接書けないため、鍵を Base64（テキスト形式）で保存しておき、起動時にデコードして元のバイト列に戻す設計。

### IV（初期化ベクトル）の長さは何に影響するか？

`GCM_IV_LENGTH = 12`（バイト = 96ビット）は NIST 推奨の AES-GCM 標準長。

IV の長さは以下に影響する：

| 項目 | 内容 |
|------|------|
| 暗号文の一意性 | 同じ鍵・平文でも IV が異なれば異なる暗号文になる。IV が被ると平文がパターン解析されるリスクがある |
| 認証タグの安全性 | GCM は改ざん検知タグを生成するが、IV が被るとタグが偽造可能になる致命的な脆弱性につながる |
| パフォーマンス | 96ビット以外だと追加のハッシュ計算が発生し非効率 |

IV は秘密にする必要はないが、**暗号化のたびにランダムに生成して使い回さない**ことが鉄則。

### @Value アノテーションの役割

```java
@Value("${app.token-enctyption-key}") String base64Key
```

- `@Value("${...}")` はアノテーション。Spring が `application.yml` の値を読み込んで変数に自動注入する
- `String base64Key` は注入された値を受け取る変数
- 引数が2つあるわけではなく、1つの引数にアノテーションが付いている形

### 共通鍵を Base64 エンコードして設定ファイルに書く理由

共通鍵は生のバイト列（バイナリ）のため、テキストファイル（application-dev.yml 等）に直接書けない。
そのため開発者が事前に Base64 エンコードした値を設定ファイルに書き、アプリ起動時にデコードして元のバイト列に戻す。

```bash
# 鍵の生成例
openssl rand -base64 32
```

```yaml
# application-dev.yml
app:
  token-enctyption-key: "aBcDeFgH..."  # Base64 エンコード済みの値
```

実運用では application.yml に直書きせず、AWS Secrets Manager や環境変数経由で渡すのがセキュア。

### System.arraycopy で IV と暗号文を結合する理由

```java
System.arraycopy(iv, 0, combined, 0, iv.length);
System.arraycopy(encrypted, 0, combined, iv.length, encrypted.length);
```

復号時に IV が必要なため、暗号文と一緒に保存する目的で1つのバイト配列に結合している。

```
combined = [ IV(12バイト) | 暗号文(nバイト) ]
```

復号時は先頭12バイトを IV として取り出し、残りを暗号文として扱う。

`System.arraycopy(src, srcPos, dest, destPos, length)` の引数：コピー元・開始位置・コピー先・貼り付け開始位置・長さ。

### Argon2 は AES-GCM と同じ共通鍵暗号方式か？

いいえ、全く異なる用途のアルゴリズム。

| | Argon2 | AES-GCM |
|---|---|---|
| 種類 | パスワードハッシュ関数 | 共通鍵暗号 |
| 方向 | 一方向（元に戻せない） | 双方向（暗号化・復号できる） |
| 用途 | パスワードの保存 | データの暗号化・復号 |

パスワードは「一致確認できればよい」のでハッシュで十分。トークンは「元の値が必要」なので暗号化が必要。

### `new SecureRandom().nextBytes(iv)` は何をやっているか？

`iv` 配列をランダムなバイト列で上書きする処理。

```java
byte[] iv = new byte[GCM_IV_LENGTH];  // 全ゼロの12バイト配列を作成
new SecureRandom().nextBytes(iv);      // ランダムな値で上書き
```

- `SecureRandom`: 暗号用途に適した予測不可能な乱数を生成するクラス（通常の `Random` より安全）
- `nextBytes(iv)`: 引数のバイト配列をランダムな値で直接上書きするメソッド（戻り値なし）

これにより暗号化のたびに異なる IV が生成され、AES-GCM の安全性が保たれる。

### IV が暗号化のたびに異なる場合、復号はどうするか？

IV は暗号文の先頭に結合して一緒に保存しているため復号できる。IV は秘密にする必要がなく、暗号文と一緒に保存して問題ない。重要なのは「使い回さないこと」だけ。

```
DBから取り出した値（Base64）
        ↓ Base64デコード
combined = [ IV(12バイト) | 暗号文(nバイト) ]
        ↓ 先頭12バイトを切り出す → iv
        ↓ 残りを切り出す → encrypted
        ↓ secretKey + iv で復号
平文（アクセストークン）
```
