# 技術Q&A - 2026-03-07

### GCMParameterSpec の認証タグとは何か？TLS/SSL の認証と同じ文脈か？

同じ「認証」という言葉だが文脈が異なる。

| | GCM の認証タグ | TLS/SSL の認証 |
|---|---|---|
| 目的 | データが改ざんされていないかを検証 | 通信相手が本物かを検証 |
| 英語 | Authentication（完全性の保証） | Authentication（身元の確認） |

GCM の「認証」は**改ざん検知付き**の意味。復号時にタグが一致しなければ例外が発生し改ざんを検知できる。TLS の「認証」は証明書を使った身元確認の話で別の文脈。

### SecurityFilterChain とは何か？HTTP リクエストが Controller に届くまでの流れ

SecurityFilterChain が先に動いて、その後 Controller が受け取る。

```
HTTPリクエスト
    ↓
SecurityFilterChain（CSRF チェック → セッション確認 → 認証チェック）
    ↓ 全フィルター通過
DispatcherServlet（Spring MVC の入口）
    ↓
Controller
```

`filterChain` メソッドは Spring 起動時に1回だけ実行され、返した `SecurityFilterChain` オブジェクトが DI コンテナに登録される。以降の全リクエストにこのオブジェクトが適用される。

### authorizeHttpRequests の書き方

```java
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/auth/login", "/api/auth/callback").permitAll()
    .anyRequest().authenticated()
)
```

- `.requestMatchers(...).permitAll()`: 指定 URL は未認証でもアクセス可
- `.anyRequest().authenticated()`: それ以外は認証必須
- `auth -> auth` は Java のラムダ式

### @Bean / @Service / @Component の違い

| | 何を指定するか | DI コンテナに登録されるもの |
|---|---|---|
| `@Service` / `@Component` | クラス | そのクラスのインスタンス |
| `@Bean` | メソッド | メソッドの戻り値のオブジェクト |

`@Bean` を使う主なケース:
- 外部ライブラリのクラス（`WebClient`, `SecurityFilterChain` など）はクラスに直接 `@Service` を付けられないため
- 生成時の設定が複雑な場合

登録タイミングはどちらも **Spring 起動時**。

コンストラクタが未定義の場合、Java がデフォルトコンストラクタ（引数なし）を自動生成し、そのコンストラクタで生成したインスタンスが DI コンテナに登録される。`@Value` によるフィールド注入はコンストラクタ実行後に Spring が行う（インスタンス生成 → フィールドへの値注入の順）。

```java
// コンストラクタ未定義でも @Value が注入された状態で DI コンテナに登録される
@Configuration
@Getter
public class FitbitConfig {
    @Value("${fitbit.client-id}")
    private String clientId;  // 起動時に application.yml の値が注入される
}

// 使う側はそのまま参照できる
@Component
@RequiredArgsConstructor
public class FitbitOAuthClient {
    private final FitbitConfig fitbitConfig;  // 注入済みの状態で渡ってくる

    public void someMethod() {
        fitbitConfig.getClientId();  // そのまま使える
    }
}
```

### Spring Security でできること（認証・認可）

Spring Security の2大責務：

| 責務 | 意味 | 例 |
|---|---|---|
| 認証（Authentication） | 誰であるかを確認する | ログイン・トークン検証 |
| 認可（Authorization） | 何をしていいかを制御する | 管理者のみアクセス可・ログイン済みのみアクセス可 |

`FilterChainProxy` が DI コンテナに登録された `SecurityFilterChain` オブジェクトを収集し、リクエストのたびにフィルターを適用する。認証ロジックをカスタマイズしたい場合は Spring Security が用意したインターフェース・抽象クラスを実装して差し込む。

```
Spring Security が用意した拡張ポイント:
    implements UserDetailsService      → フォーム認証のユーザー取得
    extends OncePerRequestFilter       → JWT 検証フィルター追加
```

### フォーム認証 vs API 認証（JWT）の違い

| | フォーム認証 | API 認証（JWT） |
|---|---|---|
| 状態管理 | サーバー側でセッション管理 | サーバー側に状態を持たない |
| 認証情報の渡し方 | Cookie（セッションID） | リクエストヘッダー（`Authorization: Bearer xxx`） |
| 向いている用途 | Web ブラウザ | モバイルアプリ・SPA・外部API |

### UserDetailsService はインターフェース・OncePerRequestFilter は抽象クラス

`UserDetailsService` はインターフェース（`implements`）。Spring Security がログイン時に DI コンテナから `UserDetailsService` 型の Bean を探して `loadUserByUsername()` を呼び出す。

`OncePerRequestFilter` は抽象クラス（`extends`）。1リクエストにつき1回だけ実行する共通処理が実装済みで、`doFilterInternal()` だけを子クラスに実装させる。

| | インターフェース | 抽象クラス |
|---|---|---|
| キーワード | `implements` | `extends` |
| 共通処理の実装 | 持てない | 持てる |
| 使いどころ | 「何ができるか」の契約 | 共通処理を持ちつつ一部を子クラスに委ねる |

### @SpringBootApplication に内包されているアノテーション

```java
@SpringBootConfiguration  // = @Configuration（設定クラスとして機能）
@EnableAutoConfiguration  // 依存関係を見て必要な設定を自動で行う
@ComponentScan            // パッケージ以下をスキャンして @Service / @Component 等を自動検出
```

`@SpringBootApplication` 1つで上記3つが一度に有効になる。
