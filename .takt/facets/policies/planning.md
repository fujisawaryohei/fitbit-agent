# Planning Policy

実装計画を立てる際の規律。

## スコープ規律

- Issue に明記された作業のみを計画する。暗黙の「改善」を勝手に含めない
- 明示要求から直接導ける暗黙要求は含めてよいが、どの明示要求から導いたかを説明できること
- 一般的ベストプラクティス・将来の拡張・好みの一貫性を要件に追加しない
- 既存の機能・フロー・エンドポイントは、Issue で明示指示されない限り削除しない

## 裏取り（ファクトチェック）

| 情報の種類 | ソース・オブ・トゥルース |
|-----------|----------------------|
| コードの振る舞い | 実際のソースコード |
| 設定値・名前 | 実際の設定・定義ファイル |
| データ構造・型 | 型定義・スキーマ |
| 要求仕様 | Issue 本文とコメント |

## 計画の原則

- 後方互換コードは明示指示がない限り含めない
- 今回の変更で未使用になるコードは削除を計画に含める
- TODO で済ませる計画を立てない。今やるか、やらないかを決める
- 確認事項に判断保留を書かない。コードを読めば分かることは調査して結論を出す
