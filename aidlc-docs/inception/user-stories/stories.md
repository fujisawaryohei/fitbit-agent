# User Stories — Fitbit Weight Management AI Agent

**構造**: Epic 階層（Epic → User Story）
**受け入れ条件フォーマット**: 箇条書き（シンプル）
**対象ペルソナ**: Ryo（ヘルスコンシャスな自己管理ユーザー）

---

## Epic 1: Fitbit 連携 & 認証

> Fitbit アカウントと安全に連携し、データを取得できる状態にする。

---

### US-01: Fitbit OAuth2 認証
**As** Ryo,
**I want** Fitbit アカウントでアプリにログインしたい,
**So that** 自分の健康データへのアクセス権を付与できる.

**Acceptance Criteria**:
- 「Fitbit と連携する」ボタンをクリックすると Fitbit の認証画面に遷移できること
- 認証を許可するとアプリに戻り「連携完了」と表示されること
- 認証を拒否した場合、「連携キャンセル」と表示されること
- 取得したアクセストークンは `.env` または安全なストレージに保存されること

---

### US-02: トークン自動リフレッシュ
**As** Ryo,
**I want** トークンが自動的に更新されてほしい,
**So that** 再ログインなしに長期間アプリを使い続けられる.

**Acceptance Criteria**:
- アクセストークン期限切れ時にリフレッシュトークンで自動更新されること
- リフレッシュに失敗した場合は再認証を促すメッセージが表示されること

---

### US-03: 連携状態の確認
**As** Ryo,
**I want** Fitbit との連携状態を UI で確認したい,
**So that** 接続が切れていることに気づいて対処できる.

**Acceptance Criteria**:
- チャット画面に「Fitbit 接続済み / 未接続」のステータスが表示されること
- 未接続の場合は「連携する」ボタンが表示されること

---

## Epic 2: 健康データ取得（LangGraph Tools）

> Fitbit API から必要な健康データを取得し、エージェントが利用できるようにする。
>
> **実装方針**: このEpicの各データ取得機能は、LangGraph の `@tool` デコレータで定義した関数としてエージェントに登録する。エージェント（ReActループ）がユーザーのリクエストに応じて必要なツールを自律的に選択・呼び出す。
>
> ```python
> from langchain_core.tools import tool
>
> @tool
> def get_steps(date: str) -> dict:
>     """指定日の歩数を Fitbit API から取得する"""
>     ...
>
> tools = [get_steps, get_weight, get_heart_rate, ...]
> agent = create_react_agent(llm, tools)
> ```

---

### US-04: 活動量データの取得
**As** Ryo,
**I want** 今日の歩数と消費カロリーをエージェントに参照してもらいたい,
**So that** 実際の活動量に基づいた提案を受けられる.

**実装**: `get_steps(date)` / `get_calories_burned(date)` ツールとして定義

**Acceptance Criteria**:
- エージェントが指定日の歩数・消費カロリーを Fitbit API から取得できること
- データが取得できない場合（未装着等）はその旨を自然言語で伝えること

---

### US-05: 体重・BMI データの取得
**As** Ryo,
**I want** 現在の体重・BMI をエージェントに参照してもらいたい,
**So that** 目標体重との差分をもとにプランを立ててもらえる.

**実装**: `get_weight(date)` ツールとして定義

**Acceptance Criteria**:
- エージェントが直近の体重・BMI ログを Fitbit API から取得できること
- 体重データが記録されていない場合はチャットで直接入力を促すこと

---

### US-06: 心拍数データの取得
**As** Ryo,
**I want** 安静時心拍数をエージェントに参照してもらいたい,
**So that** 運動強度の提案に活用してもらえる.

**実装**: `get_heart_rate(date)` ツールとして定義

**Acceptance Criteria**:
- エージェントが指定日の心拍数サマリーを Fitbit API から取得できること

---

### US-07: Fitbit 食事ログの取得
**As** Ryo,
**I want** Fitbit アプリに記録した食事の摂取カロリーをエージェントに参照してもらいたい,
**So that** 食事ログがある日はそれを考慮したプランを受けられる.

**実装**: `get_calories_in(date)` ツールとして定義

**Acceptance Criteria**:
- エージェントが Fitbit Food Log API から摂取カロリーを取得できること
- 食事ログがない日は「記録がありません」と伝え、チャット入力を促すこと

---

## Epic 3: チャット対話 & データ入力

> 自然言語でエージェントと対話し、Fitbit で補えないデータをチャットで補完できる。

---

### US-08: 自然言語チャット
**As** Ryo,
**I want** エージェントとテキストチャットでやり取りしたい,
**So that** 直感的に質問やリクエストを伝えられる.

**Acceptance Criteria**:
- Next.js のチャット画面からメッセージを送信できること
- エージェントの応答がチャット画面に表示されること
- レスポンスのストリーミング表示に対応していること

---

### US-09: チャットによる食事データ補完
**As** Ryo,
**I want** 「今日は1800kcal食べた」とチャットで入力したい,
**So that** Fitbit に食事を記録していない日でも摂取カロリーを考慮したプランを受けられる.

**Acceptance Criteria**:
- チャットで摂取カロリーを伝えるとエージェントがそれを認識して利用できること
- Fitbit の食事ログとチャット入力の両方がある場合、どちらを優先するか確認すること
- 認識した摂取カロリーをレスポンス内で明示的に提示すること

---

## Epic 4: 目標体重プランニング

> ユーザーの健康データをもとに、実現可能な目標体重達成プランを提案する。
>
> **実装方針**: プランニングロジック（カロリー計算・運動プラン生成・食事プラン生成）も LangGraph の `@tool` として定義し、エージェントが必要に応じて自律的に呼び出す。

---

### US-10: 目標体重プランの生成
**As** Ryo,
**I want** 目標体重と達成期間を伝えてプランを作ってもらいたい,
**So that** 何をどれくらいすれば良いか具体的にわかる.

**実装**: `calculate_calorie_deficit(current_weight, target_weight, weeks)` ツールとして定義

**Acceptance Criteria**:
- 目標体重・目標期間を入力するとカロリー赤字の必要量を計算できること
- 1日あたりの推奨摂取カロリーが提示されること
- 体重データが取得できない場合はチャットで入力を促すこと

---

### US-11: 食事プランの提案
**As** Ryo,
**I want** 目標達成のための食事カロリー目標を提案してもらいたい,
**So that** 何をどれだけ食べれば良いか指針が得られる.

**Acceptance Criteria**:
- 1日の推奨摂取カロリーが提示されること
- 極端な食事制限（1200kcal未満）の場合は安全上の警告が含まれること

---

### US-12: 自宅運動プランの提案
**As** Ryo,
**I want** 器具不要・自宅でできる運動プランを提案してもらいたい,
**So that** ジムに行かなくても具体的に実践できる.

**実装**: `generate_home_workout_plan(calorie_target, fitness_level)` ツールとして定義

**Acceptance Criteria**:
- 種目名・回数・セット数が提示されること
- 各種目の消費カロリー目安が含まれること
- 週次スケジュール（例：月水金に実施）が提案されること
- 各種目のフォーム説明が含まれること
- 初心者向けのバリエーション（負荷を下げた代替動作）が含まれること

---

### US-13: 進捗確認
**As** Ryo,
**I want** 今週の活動量と体重変化の進捗を確認したい,
**So that** 目標に対してどの程度進んでいるか把握できる.

**実装**: `get_weekly_progress(start_date, end_date)` ツールとして定義（内部で複数の Fitbit ツールを集約）

**Acceptance Criteria**:
- 「今週の進捗を教えて」と聞くと週次サマリーが返ってくること
- 歩数・消費カロリー・体重変化が含まれること
- 目標に対する評価コメント（良好 / 要改善 等）が含まれること

---

## Epic 5: メモリ管理

> 会話の文脈と長期的なユーザー情報を保持し、毎回説明し直す手間をなくす。

---

### US-14: ショートタームメモリ（In-Context Memory）
**As** Ryo,
**I want** 同じ会話セッション内で文脈が引き継がれてほしい,
**So that** 「さっき言った目標体重に基づいて」という流れで話せる.

**実装**: LangGraph の `MessagesState`（messages リスト）を state として保持する in-context memory

**Acceptance Criteria**:
- 同一セッション内での会話履歴がエージェントに渡ること
- 過去の発言を参照した応答が生成されること

---

### US-15: ロングタームメモリ（セマンティックメモリ）
**As** Ryo,
**I want** 目標体重・食事の好み・運動の好みを次回セッションでも覚えていてほしい,
**So that** 毎回ゼロから説明しなくても自分に合った提案を受けられる.

**実装**: ベクトルストア（Chroma または FAISS）を使ったセマンティック検索によるロングタームメモリ。重要情報（目標・好み・過去のプラン）をエンベディングして保存し、セッション開始時に関連情報を検索・取得してコンテキストに注入する。

**Acceptance Criteria**:
- 目標体重・期間などの重要情報がベクトルストアに保存されること
- 次回セッション開始時に関連情報を検索してコンテキストに注入できること
- 「目標体重を58kgに変更して」という更新リクエストが反映されること
