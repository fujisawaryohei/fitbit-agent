# マルチプロバイダ相互レビューがなぜ良いとされるのか

複数のAIプロバイダ／モデルを使い、一方の出力を別モデルにレビューさせる構成
（例: Claude が計画 → Codex がレビュー、Codex が実装 → Claude がレビュー）が
推奨される理由と、その限界を整理したもの。

関連: [takt-rules-routing.md](./takt-rules-routing.md)
対象ワークフロー: `.takt/workflows/issue-driven-dev.yaml`

---

## 結論サマリ

> 本質は **「誤りの相関を下げて検出力を上げる」** こと。
> self-preference bias（自己選好）を避け、モデルごとの盲点を補完し、ベンダー依存も減らせる。
> 代償はコストとレイテンシで、両モデルが共通して苦手な領域までは消せない。

---

## 1. 誤りの「相関」が下がる（最大の理由）

同じモデルに自分の出力をレビューさせると、**生成時と判定時で同じ偏り・同じ盲点を共有する**ため、
自分の間違いに気づけない。

- モデルは「自分が尤もらしいと判断した出力」を生成する。同じモデルが評価すると、また「尤もらしい」と判定しがち（self-consistency bias）。
- 学習データ・アーキテクチャ・RLHF の方針が異なる別モデルは、**誤りの分布が異なる**。片方が見落とす種類のバグを、もう片方が捉えやすい。
- レビューの検出力は「2つの判定が独立しているほど」高くなる。同一モデルは判定が強く相関し独立性が低いため、検出力が上がらない。

## 2. self-preference bias（自己選好）の緩和

LLM には **「自分が書いた文章を高く評価する」傾向**が報告されている。

- 同一モデルの自己レビューは「Approve」に流れやすく、ループが形骸化する。
- 別プロバイダにレビューさせると身内びいきが効かず、**承認のハードルが正当に保たれる**。
- これは TAKT の設計哲学 **"Separation Improves Quality"**（実装者が自分自身をレビューしてはいけない）を、モデルレベルでも徹底する発想。

## 3. モデルごとの得意領域の補完

| 観点 | 効果 |
|------|------|
| 一方が計画・抽象設計に強く、他方が実装・エッジケース検出に強い | 工程ごとに適材を当て、レビューで相互補完 |
| プロバイダ固有の典型的失敗（特定 API の幻覚など） | 別系統のモデルが検出しやすい |

→ `issue-driven-dev.yaml` で **Claude=計画/レビュー、Codex=実装** と振っているのは、
この「役割 × モデルの相性」を使った設計。

## 4. 単一ベンダー依存リスクの低減

- 1 社のモデルの癖・障害・レート制限・仕様変更に全工程が引きずられない。
- TAKT に `rate_limit_fallback`（レート制限時に別プロバイダへ切替）があるのも、この冗長性思想の延長。

---

## 注意点（万能ではない）

1. **コスト・レイテンシ増** — 工程ごとに別 API、さらに判定用 conductor 呼び出しも乗る。
2. **誤りが消えるわけではない** — 両モデルが共通して苦手な領域（曖昧な要件、ドメイン固有知識）は、何回ループしても検出されない。独立といっても完全独立ではない。
3. **「指摘のための指摘」** — 別モデルだと文化差で好みベースの差し戻しが増えることがある。
   → `policies/review.md` で「好みは差し戻し理由にしない」と縛るのが対策。
4. **本質は "独立性" であって "別ベンダーであること" そのものではない** — 同一モデルでも temperature やペルソナ・観点を大きく変えれば一定の多様性は出る。別プロバイダはその独立性を最も手軽かつ強く確保する手段、という位置づけ。

---

## 実務上の使い分け

| ケース | 方針 |
|--------|------|
| 品質が重要な工程 | マルチプロバイダ相互レビューを入れる |
| 軽微なタスク | 単一モデル（`*-mini` 系など）で回し、コストを抑える |
| レビューの形骸化を防ぎたい | レビュー側を別プロバイダにし、policy で好み差し戻しを禁止する |

---

## 用語

| 用語 | 意味 |
|------|------|
| self-preference / self-enhancement bias | LLM が自分の生成物を他より高評価しやすい傾向 |
| self-consistency bias | 生成時と評価時で同じ判断基準を共有し、自分の誤りを見逃す傾向 |
| 誤りの相関 | 2 つの判定が同じ箇所で同時に間違う度合い。低いほどレビューの検出力が高い |
