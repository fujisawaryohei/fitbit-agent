## 2. バックエンド構成

> 使用する言語・フレームワークに合わせてディレクトリ名・ファイル名を調整すること。
> 以下は一般的なレイヤードアーキテクチャの例。

```
{バックエンドディレクトリ}/
├── {ソースルート}/
│   ├── {エントリポイント}
│   ├── {controller相当}/     # HTTPリクエスト受付・レスポンス返却
│   ├── {service相当}/        # ビジネスロジック
│   ├── {repository相当}/     # DBアクセス層
│   ├── {domain相当}/         # エンティティ・値オブジェクト
│   ├── {client相当}/         # 外部APIクライアント
│   ├── {dto相当}/            # リクエスト・レスポンス データ構造
│   ├── {config相当}/         # フレームワーク設定
│   └── {exception相当}/      # 例外クラス・エラーハンドラー
├── {リソースディレクトリ}/
│   ├── {設定ファイル}        # アプリケーション設定（例: application.yml, settings.py）
│   └── {DBマイグレーション}/ # マイグレーションファイル（例: db/migration/）
├── {テストディレクトリ}/     # テストコード（本体と同じパッケージ構造）
└── {ビルド定義ファイル}      # 例: pom.xml, build.gradle, pyproject.toml, package.json
```

### 配置ルール

| モジュール種別 | 配置先 | 命名規則 |
|-------------|--------|---------|
| リクエスト受付 | `{controller相当}/` | `{機能名}{Controller相当クラス名}` |
| ビジネスロジック | `{service相当}/` | `{機能名}{Service相当クラス名}` |
| DBアクセス | `{repository相当}/` | `{エンティティ名}{Repository相当クラス名}` |
| ドメインモデル | `{domain相当}/` | `{テーブル名・エンティティ名}` |
| 外部APIクライアント | `{client相当}/` | `{サービス名}{Client相当クラス名}` |
| 入出力データ構造 | `{dto相当}/` | `{機能名}Request` / `{機能名}Response` |

## 3. フロントエンド構成

> 使用するフレームワーク（React, Vue, Svelte 等）に合わせてディレクトリ名・ファイル拡張子を調整すること。

```
{フロントエンドディレクトリ}/
├── {ソースルート}/
│   ├── {エントリポイント}
│   ├── {pages相当}/      # ページ単位のコンポーネント・ビュー（ルーティング対応）
│   │   └── {機能名}/
│   ├── {components相当}/ # 再利用可能な UI コンポーネント
│   │   └── {カテゴリ}/
│   ├── {状態管理}/       # 状態管理（store / hooks / composables 等）
│   ├── {api相当}/        # APIクライアント・エンドポイント定義
│   ├── {types相当}/      # 型定義（TypeScript 等）
│   ├── {utils相当}/      # ユーティリティ関数
│   └── {router相当}/     # ルーティング設定
├── {静的ファイル}/
└── {パッケージ定義ファイル}  # 例: package.json
```

### 配置ルール

| ファイル種別 | 配置先 | 命名規則 |
|------------|--------|---------|
| ページ・ビュー | `{pages相当}/{機能名}/` | `{機能名}{Page/View相当}` |
| 共通コンポーネント | `{components相当}/{カテゴリ}/` | `{コンポーネント名}` |
| API定義 | `{api相当}/` | `{機能名}{Api/Service相当}` |
| 型定義 | `{types相当}/` | `{機能名}.types.{拡張子}` |

## 5. 配置ルール（共通）

- **新機能のファイルは必ず既存のレイヤー構造に従う**
- **レイヤーをまたぐ依存は上位 → 下位の一方向のみ**（リクエスト受付 → ビジネスロジック → DBアクセス）
- **外部APIクライアントは専用ディレクトリに集約し、ビジネスロジック層から呼び出す**
- **テストファイルは本体と同じディレクトリ構造をテスト用ルート以下に再現する**
