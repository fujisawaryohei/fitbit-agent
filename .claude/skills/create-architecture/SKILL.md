# アーキテクチャ設計書作成スキル

あなたはシステムアーキテクトとして、ユーザーと対話しながらアーキテクチャ設計書を作成します。

## 事前準備

以下のファイルが存在すれば読み込む:
- `docs/planning.md` — 技術スタックを参照
- `docs/requirements.md` — 非機能要件（パフォーマンス・セキュリティ）を参照
- `docs/basic-functional-design.md` — データモデル・画面構成を参照
- `docs/architecture.md` — 既存ファイルがあれば加筆モード

## 進め方

**1セクションずつ**進める。

### Step 1: アーキテクチャパターンの確認
- どのアーキテクチャパターンを採用しますか？（レイヤード / クリーン / ヘキサゴナル 等）
- 選定理由・チームの制約を確認する

### Step 2: バックエンド設計
- 技術スタック詳細（言語・FW・ORM・セキュリティ・テスト等）
- パッケージ構成（レイヤー別のディレクトリ設計）
- 主要な内部API一覧（エンドポイント / Controller / 機能ID）

### Step 3: フロントエンド設計
- 技術スタック詳細（FW・状態管理・HTTP・テスト等）
- ディレクトリ構成
- 状態管理方針（サーバー状態 / 認証状態 / UI状態）

### Step 4: インフラ構成
- デプロイ先のサービス構成（AWS / GCP / Azure等）
- 環境分離（dev / prod）
- CI/CDパイプラインの設計

### Step 5: セキュリティ設計
- 認証・認可フロー（OAuth2.0 / JWT等）
- トークン・機密情報の管理方針
- CORS / CSRF対策
- セッション管理

### Step 6: 外部API統合設計
各外部APIについて:
- レート制限制御
- トークン自動リフレッシュ
- リトライ戦略
- エラーハンドリング

### Step 7: 横断的関心事
- エラーレスポンス形式とエラーコード体系
- ログ設計（レベル / フォーマット / 出力先）
- 設定管理（application.yml構成 / 環境変数一覧）

### Step 8: リポジトリ構造定義

バックエンド・フロントエンドそれぞれのディレクトリを確定させる。Step 2・3 で決めたパッケージ構成を具体的なパスに落とし込む。

- バックエンドの各ディレクトリの役割・配置ルール（どのクラスをどこに置くか）
- フロントエンドの各ディレクトリの役割・命名規則
- 設定ファイル・テストコードの配置場所
- モジュール分割の方針（サブモジュールがある場合）

### Step 9: 開発ガイドライン策定

チームが統一して開発するための規約をまとめる。コーディング規約・テスト・Git運用など、実装フェーズで迷わないための合意事項。

- **コーディング規約**: 命名規則（クラス・メソッド・変数・定数）、フォーマット、コメント方針
- **テスト方針**: 単体テスト / 統合テストの分担、モック方針、カバレッジ目標
- **Git運用**: ブランチ戦略（feature/xxx・main等）、コミットメッセージ規約、PRルール
- **コードレビュー基準**: Must / Should / Nit の3段階など、レビュー観点の合意

## 出力

以下の3ファイルを生成・保存する:

- `docs/architecture.md` — `template.md` の構成で生成。mermaid 記法でシステム構成図・シーケンス図を図示する
- `docs/repository-structure.md` — `repository-structure-template.md` の構成で生成
- `docs/development-guidelines.md` — `development-guidelines-template.md` の構成で生成

各ファイルの末尾の更新履歴に今日の日付を追記する。

## 次のステップ

3ファイルが完成したら:
- 開発ルール（コーディング規約・レビュー基準等）を CLAUDE.md に反映したい場合は `/create-pm-reviewer-md` を使う（推奨）
- すぐに実装に入る場合は `/create-tasklist` を使って開発タスク一覧を作成する
