## 3. Git運用

### 3.1 ブランチ戦略

```
main               ← 本番リリース済みコード
  └── feature/{チケットID}-{概要}   ← 機能開発（例: feature/TASK-010-auth）
  └── fix/{チケットID}-{概要}       ← バグ修正
  └── chore/{概要}                  ← 設定・依存関係更新等
```

### 3.2 コミットメッセージ規約

**形式**: `{type}: {summary}`

| type | 用途 |
|------|------|
| `feat` | 新機能追加 |
| `fix` | バグ修正 |
| `refactor` | リファクタリング |
| `test` | テスト追加・修正 |
| `chore` | ビルド・依存関係・設定変更 |
| `docs` | ドキュメント更新 |

**ルール**:
- コミットメッセージは**英語**で記述する
- summary は命令形・現在形（"Add", "Fix", "Update"）
- 1コミット = 1つの論理的変更

### 3.3 PRルール

- ブランチ名は `feature/TASK-xxx-{説明}` 形式
- PR タイトルはコミット規約に準拠
- レビュワーを最低1名アサインする
- CI（テスト・ビルド）が通過していること

## 4. コードレビュー基準

レビューコメントは以下の3段階で分類する:

| 分類 | 意味 | 対応 |
|------|------|------|
| **Must** | マージ前に必ず修正 | バグ・セキュリティ脆弱性・設計違反 |
| **Should** | 推奨修正（理由を説明する） | 可読性・保守性・テスト不足 |
| **Nit** | 軽微な指摘（対応任意） | 命名・フォーマット・コメント |

### レビュー観点

- **正確性**: 要件・受入条件を満たしているか
- **設計**: アーキテクチャ・レイヤー構成に違反していないか
- **セキュリティ**: 入力バリデーション・認証・認可が適切か
- **テスト**: 主要なケースがテストされているか
- **可読性**: 命名・コメントが適切か
