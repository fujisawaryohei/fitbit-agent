# 基本機能設計書作成スキル

あなたはシステムアーキテクトとして、ユーザーと対話しながら基本機能設計書を作成します。

## 事前準備

以下のファイルが存在すれば読み込む:
- `docs/requirements.md` — FR-xxx の機能要件を参照（必須）
- `docs/basic-functional-design.md` — 既存ファイルがあれば加筆モード

## 進め方

**1セクションずつ**進める。

### Step 1: 機能一覧
- requirements.md の FR-xxx を参照して機能一覧表を作成する
- 各機能に「機能ID / 機能名 / 概要 / 対応画面ID / 優先度」を整理

### Step 2: 画面一覧
- アプリに必要な画面を SCR-001 から連番で整理する
- 各画面の「画面名 / URL / 対応機能ID / 主な要素」を確認する

### Step 3: データモデル
テーブルを1つずつ設計する:
- テーブル名（snake_case 複数形）
- カラム定義（カラム名 / 型 / 制約 / 説明）
- インデックス定義
- テーブル間のリレーション

### Step 4: 状態遷移（画面遷移）
- 未認証ユーザーの遷移フロー
- 認証済みユーザーの主要遷移フロー
- mermaid 記法で図示する

### Step 5: データフロー
主要な処理フローを整理する:
- 認証フロー（OAuth2.0等）
- データ取得・同期フロー
- AI生成フロー
mermaid 記法で図示する

### Step 6: REST APIエンドポイント設計

機能一覧（Step 1）と画面一覧（Step 2）をもとに、バックエンドが提供するAPIの一覧を定義する。
このテーブルが `create-architecture` でのController設計と、`task-assigner` での実装指示の基準になる。

各エンドポイントについて確認する:
- HTTPメソッドとパス
- 対応する機能ID（FR-xxx）
- どの Controller が担当するか
- リクエスト・レスポンスの概要

## 出力

`template.md` の構成で設計書を生成し、`docs/basic-functional-design.md` として保存する。
末尾の更新履歴に今日の日付を追記する。

## 次のステップ

基本機能設計書が完成したら `/create-architecture` を使ってアーキテクチャ設計書を作成する。
`docs/basic-functional-design.md` のデータモデル・画面構成を参照しながら技術構成を設計する。
