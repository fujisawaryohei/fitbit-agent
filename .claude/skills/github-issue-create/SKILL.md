# GitHub Issue 作成スキル

## 概要

ユーザーの「やりたいこと」から出発し、仕様策定・プランニング・TODO化を経て GitHub Issue を作成し Project に追加する。

---

## Step 0: 事前確認

スキル起動時に以下を確認する:

```bash
gh auth status
gh project list --owner <owner>
```

- 認証が通っていない場合: `gh auth login` を案内して終了
- project スコープがない場合: Classic PAT での再認証を案内して終了
- Project が1つだけの場合: 自動選択してよいか確認する

---

## Step 1: やりたいことのヒアリング

ユーザーに以下だけを聞く:

> 「何をやりたいですか？背景・問題点も含めて教えてください。」

回答を受け取ったら Step 2 へ進む（この時点では仕様の深掘りはしない）。

---

## Step 2: issue-planner エージェントによる仕様策定・プランニング

`issue-planner` エージェントにユーザーの回答を渡し、以下を出力させる:

- 種別（feat / fix / chore / docs）
- タイトル（要約）
- 概要
- 仕様（箇条書き）
- TODO（チェックボックス形式の実装ステップ）
- 参照ドキュメント
- 完了条件（チェックボックス形式の動作確認観点）

エージェントが確認質問をした場合はユーザーに転送し、回答を受け取ってから再度エージェントに渡す。

---

## Step 3: ユーザー確認・修正

エージェントの出力をユーザーに見せて確認を取る:

> 「以下の内容で Issue を登録してよいですか？修正があれば教えてください。」

```
種別: {種別}
タイトル: {タイトル}

## 概要
{概要}

## 仕様
{仕様}

## TODO
- [ ] 実装ステップ 1
- [ ] 実装ステップ 2
...

## 参照ドキュメント
{参照ドキュメント}

## 完了条件
- [ ] 動作確認条件 1
- [ ] 動作確認条件 2
...
```

修正依頼があれば内容を更新して再確認する。「OK」「問題ない」等の承認を得たら Step 4 へ進む。

---

## Step 4: GitHub Issue 作成

承認された内容で Issue を作成する:

```bash
gh issue create \
  --title "{種別}: {タイトル}" \
  --body "## 概要
{概要}

## 仕様
{仕様}

## TODO
{TODOをチェックボックス形式で}

## 参照ドキュメント
{参照ドキュメント}

## 完了条件
{完了条件をチェックボックス形式で}"
```

---

## Step 5: Project への追加

Issue 作成で返却された URL を使って Project に追加する:

```bash
gh project item-add {project番号} --owner {owner} --url {issue_url}
```

---

## Step 6: 完了・次の Issue 確認

```
✅ Issue を登録しました
- タイトル: {種別}: {タイトル}
- URL: {issue_url}
- Project: {project番号} に追加済み

続けて別の Issue を登録しますか？
```

「はい」の場合は Step 1 に戻る。「いいえ」の場合はスキルを終了する。
